커밋 17개, z-index 버그 4번, 결제 수단 3중 구조, 빌드 로그 자동화까지 — 전부 Claude Code와 단둘이 해결했다.

나는 1인 개발자다. 사주(四柱, Four Pillars of Destiny) AI 리포트를 판매하는 앱을 만들고 있다. 한국 전통 명리학을 AI로 해석해 PDF 리포트로 팔겠다는 아이디어다. 한국어, 일본어, 영어, 동남아 5개 언어까지 총 8개 로케일을 지원한다. 결제는 Toss(한국), Stripe(글로벌), 거기에 이번 주 Paddle까지 추가했다.

이 글은 런칭 직전 2주 동안 실제로 어떤 삽질이 있었고, AI가 그 삽질을 어떻게 해결했는지에 대한 기록이다. 코드 자랑이 아니라 작업 방식 공개다. 프롬프트, 커밋 해시, 삽질 횟수 전부 있다.

## 배경: 화면에 아무것도 안 보이면 가장 먼저 의심해야 할 것

작업이 시작된 건 단순한 불만이었다. CSS `body::before`와 `body::after`로 오로라 효과와 별 배경을 만들어 뒀는데, 이게 60fps 애니메이션에서 버벅댔다. `globals.css`에 무려 51줄이 쌓여 있었고, 유지보수도 힘들었다. 이미 프로젝트에는 Three.js `CosmicBackground` 컴포넌트가 존재했다 — 별자리 선을 그리는 기능까지 있는, 훨씬 정교한 구현체였다. 그냥 CSS를 걷어내고 Three.js로 전환하면 되는 일처럼 보였다.

실제로 해보니 전혀 아니었다.

Three.js 캔버스는 분명히 DOM에 붙어 있었다. `position: fixed`로 설정돼 있었고 크기도 맞았다. 그런데 화면에 아무것도 보이지 않았다. 개발자 도구를 열어도 캔버스 요소는 있는데 렌더링이 안 됐다. 처음엔 WebGL 초기화 문제라고 생각했다. 틀렸다.

Claude에게 던진 첫 프롬프트는 이랬다.

> "canvas가 있는데 Three.js 배경이 화면에 안 보인다. `globals.css`와 `CosmicBackground.tsx`를 같이 봐줘."

Claude가 찾아낸 건 CSS reset이었다. `* { max-width: 100% }` 룰이 `canvas` 요소에도 적용되면서 캔버스의 실제 렌더링 크기를 죽이고 있었다. 첫 커밋(`fcff7ec`)은 캔버스를 reset 룰에서 제외하는 한 줄짜리 수정이었다.

고쳤다. 여전히 안 보였다.

## 딥다이브 1: z-index는 음수가 되는 순간 html 뒤로 사라진다

두 번째 원인은 `body` 배경색이었다. `body`가 불투명한 네이비(`#0a0b2a`)로 설정돼 있었고, Three.js 캔버스가 `z-index: -2`로 body 뒤에 깔려 있었다. 캔버스는 거기 있었지만 body가 덮고 있었다. `html` 요소에 기본 배경을 넘기고 `body`를 `background: transparent`로 바꿨다(`96a5555`). CSS aurora/star 코드 51줄도 이 시점에 일괄 삭제했다(`1ec4fcb`). 그러니까 처음 목표였던 CSS 걷어내기는 두 번째 커밋에서야 이루어진 것이다.

여전히 완전하지 않았다.

세 번째 문제가 가장 오래 걸렸다. `CosmicBackground`의 캔버스가 `z-index: -2`에서 `z-index: -1`로 조정됐는데도 콘텐츠 뒤로 숨었다. 이 시점에 Claude가 지적한 게 z-index 스택 컨텍스트였다.

> "z-index -2는 html 루트 스택 컨텍스트 아래로 떨어진다. 캔버스를 0/1로 올리고 콘텐츠(`main`, `footer`)에 z-index: 2를 줘라."

음수 z-index가 `html` 요소보다 아래로 가버리는 건 CSS 스펙 상 맞는 동작이다. 그런데 실제로 삽질하기 전엔 직관적으로 떠오르지 않는다. 캔버스를 `z-index: 0`으로, 콘텐츠 레이어를 `z-index: 2`로 설정하고 해결됐다(`052de6e`). 같은 커밋에서 CTA 버튼의 마그네틱 마우스 팔로우 효과도 제거했다. 만들 때 공을 들였지만 복잡도 대비 UX 효과가 없다고 판단했다.

정리하면 Three.js 배경 하나를 띄우는 데 독립적인 원인 세 가지가 있었다. CSS reset, body 불투명도, z-index 음수. 하나씩 고쳐야 했기 때문에 커밋이 4개로 쪼개졌다. 사실 이게 AI와의 작업에서 중요한 포인트다 — 원인이 복합적일 때 Claude가 "이것저것 다 바꿔볼게요"로 접근하면 오히려 더 헷갈린다. 한 번에 한 가지 가설을 검증하게 유도해야 한다. 이번엔 그게 잘 됐다.

| 커밋 | 원인 | 수정 |
|------|------|------|
| `fcff7ec` | CSS reset이 canvas 죽임 | canvas를 max-width reset 제외 |
| `96a5555` | body 불투명으로 canvas 가림 | body transparent, html에 배경 이동 |
| `1ec4fcb` | CSS aurora/star 코드 잔존 | globals.css 51줄 삭제 |
| `052de6e` | z-index 음수가 html 아래로 | 0/1/2로 재설정 |

## 딥다이브 2: Paddle을 기존 코드 건드리지 않고 추가하는 법

Three.js 작업과 거의 동시에 Paddle 통합이 진행됐다. Stripe가 글로벌 결제를 담당하고 있었는데, Paddle을 1차 결제 수단으로 올리고 싶었다. Paddle은 merchant of record 모델이라 부가세나 세금 처리를 알아서 해준다. 동남아, 일본 고객에게는 현실적으로 Stripe보다 전환율이 높다.

제약 조건을 먼저 명시했다. "기존 Stripe/Toss 코드를 건드리지 않고, 런타임 환경변수 하나로 전환 가능하게" — 이 요구사항을 그대로 프롬프트에 넣었다. Claude가 설계한 건 `effectiveProvider` 로직이었다. `NEXT_PUBLIC_PAYMENT_PROVIDER=paddle` 환경변수 하나가 결제 흐름 전체를 분기한다. 기존 Stripe/Toss 라우트는 손대지 않고 `/api/checkout/paddle/create`와 `/api/checkout/paddle/webhook` 두 라우트가 새로 생겼다.

배포 직전에 Playwright로 8개 로케일 전부 E2E 스모크 테스트를 돌렸다. `localhost:3000`에서 `typecheck`와 `next build`가 통과하는 것, 그리고 paywall 접근이 전부 PASS하는 것을 확인하고 커밋했다. 런칭 감사 보고서(`CLAUDE_AUDIT_LAUNCH_READINESS_2026-03-05.md`)도 이 시점에 Claude가 작성했다. 레이트 리밋 5회 한도 실측, P0 결함 없음 확인. 보고서를 Claude에게 쓰게 할 때 조건을 달았다 — "추측 없이, 모든 근거에 파일 경로 또는 실행 결과를 명시해라." 그렇게 하지 않으면 그럴싸하지만 실제로는 아무것도 검증하지 않은 문서가 나온다.

## 딥다이브 3: Claude가 push하면 Claude가 빌드 로그를 쓴다

이번 작업에서 가장 재밌었던 부분이다. 자동화 파이프라인.

목표는 간단했다. git push를 하면 Claude CLI가 이번 작업의 빌드 로그를 자동 생성하고, GitHub Actions가 포트폴리오 레포와 블로그 레포로 동기화한다. 지금 읽고 있는 이 글도 그 파이프라인의 산출물이다.

프롬프트는 이랬다.

> "husky pre-push 훅에서 `claude -p` 로 빌드 로그를 생성하고, GitHub Actions에서 포트폴리오 레포와 블로그 레포로 자동 동기화해줘. 커밋 5개 이상이면 블로그 초안도 만들어줘."

핵심 설계 결정은 타임아웃이었다. Claude CLI가 무한정 실행되면 push 자체가 멈춰버린다. `gtimeout 180`으로 제한을 걸고, 타임아웃이 나도 hook이 블로킹하지 않도록 `|| echo "⚠️ 스킵"`으로 처리했다. 처음에 `timeout` 명령어가 macOS에 기본 내장이 아니라는 것을 몰랐다. `gtimeout`은 `brew install coreutils`로 설치해야 한다. 두 번째 삽질은 Claude CLI 경로였다. 상대 경로가 husky hook 환경에서 안 먹혔다. `/Users/jidong/.local/bin/claude` 풀패스로 고쳤다.

GitHub Actions 쪽에서는 Vercel 배포가 Next.js 프레임워크 preset을 못 잡는 문제가 있었다. `vercel.json`에 `framework: "nextjs"`를 명시하고, 루트 디렉토리를 `apps/web`으로 설정하고, 환경변수를 맞추는 과정에서 커밋이 5개 더 쌓였다. 전형적인 배포 삽질이지만 `fix(deploy):` 컨벤션으로 기록은 깔끔하게 남겼다.

마지막으로 husky pre-push 타임아웃을 60초에서 180초로 늘렸다(`2d71d3a`). Claude CLI가 빌드 로그를 생성하는 데 시간이 더 필요하기 때문이다. 60초면 복잡한 작업 기록 생성이 중간에 잘린다.

파이프라인 완성 이후로 push 한 번이면 이렇게 된다. Claude CLI가 git diff와 commit log를 읽고 한국어/영어 빌드 로그를 `build-logs/` 폴더에 저장한다. GitHub Actions가 트리거되면 포트폴리오 레포와 블로그 레포에 빌드 로그가 자동으로 올라간다. 블로그 초안은 별도 스킬로 호출하면 된다 — 지금 이 글이 그 결과다.

## 결과: 숫자로 정리

| 항목 | Before | After |
|------|--------|-------|
| 배경 CSS | globals.css 51줄 (aurora + star) | 0줄 (Three.js 이관) |
| 결제 수단 | Stripe + Toss | Stripe + Toss + Paddle (환경변수 전환) |
| 레이트 리밋 | 없음 | 5회/일 (LLM 엔드포인트, IP 기반) |
| E2E 커버리지 | 0 | 8개 로케일 × 핵심 플로우 |
| 빌드 로그 | 수동 작성 | push 시 Claude CLI 자동 생성 |
| 커밋 수 | — | 17개 (이번 사이클) |
| husky timeout | 60s | 180s pre-push / 240s fallback |

z-index 4번 디버깅이 가장 시간을 잡아먹었다. 원인이 3개였고 독립적이라 순서대로 고쳐야 했다. Claude가 각각의 원인을 정확히 짚긴 했는데, 처음엔 "이게 왜 안 보이냐"라는 너무 열린 질문을 던졌다. 나중엔 "현재 가설은 X다, 이걸 먼저 검증해라"로 좁혀서 물어봤다. 열린 질문보다 가설 기반 질문이 훨씬 빠르다.

Paddle 통합은 제약 조건을 먼저 명시한 게 주효했다. "기존 코드 건드리지 않고"라는 조건 하나가 Claude의 설계 방향 전체를 결정했다. 그 조건 없이 시작했으면 Stripe 코드를 리팩토링하는 방향으로 갔을 것이고, 건드리지 않아도 되는 코드를 건드리게 됐을 것이다.

자동화 파이프라인은 가장 잘 한 결정이다. 빌드 로그를 수동으로 쓰다 보면 미루게 된다. 글쓰기 자체가 귀찮은 게 아니라 "나중에 써야지"가 되는 순간 영원히 안 쓰게 된다. push 직후에 자동으로 생성되니까 늦어질 이유가 없다. 퀄리티가 완벽하지 않더라도 기록이 있다는 게 없는 것보다 훨씬 낫다.

## 1인 개발자가 AI를 쓰는 방식

나는 Claude에게 코드를 완성시키지 않는다. 정확히는 — 완성본을 한 번에 요청하지 않는다. 제약 조건을 먼저 정의하고, 가설을 검증하는 단계를 쪼개고, 완료 기준을 명시한다. "Paddle을 추가해줘"와 "기존 Stripe/Toss 코드 건드리지 않고, 환경변수 하나로 전환 가능하게 Paddle을 추가해줘"는 전혀 다른 결과를 만든다.

자동화는 반복되는 일에만 투자한다. 빌드 로그 생성이 반복될 것이 명확했기 때문에 파이프라인을 만들었다. 하지만 이 파이프라인 자체를 만드는 데도 몇 가지 삽질이 있었다 — macOS에 `timeout`이 없다는 것, Claude CLI 경로가 환경마다 다르다는 것. 이런 것들은 검색으로 해결할 수도 있지만 Claude에게 에러 메시지를 던지면 빠르다.

사주 앱은 아직 첫 유료 전환을 달성하지 못했다. Paddle 웹훅 실 결제 테스트가 남아 있고, ko 로케일 링크 prefix 정책도 확정되지 않았다. 그런데 런칭 준비는 됐다. 8개 로케일 E2E 통과, P0 결함 없음, 결제 3중 구조, 레이트 리밋까지 — 코드 상태로만 보면 나갈 수 있다.

다음 편에서는 첫 유료 전환 이후, 또는 전환이 안 나왔을 때의 분석을 쓰겠다. 어느 쪽이든 공개한다.

코드 레포는 비공개지만 아키텍처나 프롬프트 전략에 대한 질문은 댓글에 남겨도 된다. 같은 방식으로 프로젝트를 진행 중인 사람이 있다면 서로 공유할 게 많을 것 같다.
